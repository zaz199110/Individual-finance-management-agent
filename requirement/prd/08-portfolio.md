## 8. 模块四：持仓分析

### 模块说明

| 项 | 说明 |
|----|------|
| **做什么** | 录入/更新 **国内公募** 持仓 → 组合诊断与收益测算；报告正文 **必含本次分析的当前持仓明细快照**（§8.4.2 · **PORT-HOLDINGS-IN-REPORT-01**）；**可选** 对照 §7 **已发布**《资产配置方案》报告 → 《持仓分析报告》 |
| **输入依据** | **持仓**（`holdings_versions` 当前态）+ **可选** §7 **`allocation_plans`**（完善组 · **已发布资产配置方案报告** · step=2）— **不** 在此生成新资产配置方案 |
| **不做什么** | 预置 demo 持仓；无持仓出报告；**不** 解析 PDF/Excel/Word 对账单文件（**PORT-NO-FILE-01** · 仅 **图片 + 手输**）；**不** 替代 §7 做大类/选基方案设计（**PORT-NO-PLAN-DESIGN-01**）；**不** 用未确认规划书草稿做对照（**PORT-PLAN-PUB-01**）；**不** 反写/自动更新资产配置方案（**PORT-NO-PLAN-WRITE-01**） |
| **与 §7 分工** | §8 **读持仓**、**对照** §7 已确认方案、做偏离诊断与再平衡；§7 **不读持仓** — 两模块 **单向依赖**：§7 产出方案 → §8 可选用方案对照 |
| **入口** | Tab **持仓分析**（可上传持仓/对账单截图 · **单次最多 20 张** · PORT-VISION-BATCH-01） |
| **完成标志** | 《持仓分析报告》确认发布（§8.4）；可开定时持仓分析（§4.2） |
| **编码锚点** | `holdings_versions`；报告骨架 **PORT-01** · [portfolio spec §4](../docs/samples/portfolio-analysis-report-spec.md) |

### 8.1 当前持仓面板与录入（PORT-PANEL-01 · P0）

#### 8.1.2 首次 vs 再次进入

| 模式 | 条件 | 行为 |
|------|------|------|
| **A · 首次** | 无 `is_current=true` 或明细为空 | 引导录入（文字 / 截图）；补齐必填项 → 确认卡 → 入库 |
| **B · 再次** | 存在 `is_current=true` 且 ≥1 条明细 | **默认展示「当前持仓」Tab 或对话首句带出摘要**；助手 **一句询问**是否仍准确；**不强制**点「确认无误」；无变更可直接说「**重新分析**」；有调仓 → §8.2 |

#### 8.1.3 三种录入方式（PORT-INPUT-01 · P0）

> **本期不做** 聊天区上传 PDF/Excel/Word 并解析（**PORT-NO-FILE-01**）。对账单请用户 **截图** 上传，与全站 §0.3 一致，**省实现成本、省 Token**，且与用户「手机 App 截屏」习惯一致。

| # | 方式 | 用户怎么做 | 系统怎么做 |
|---|------|------------|------------|
| **① 手输** | 打字描述持仓 | `holdings_propose`（`source=manual`）→ 确认卡 |
| **② 持仓页截图** | 上传 App **持仓列表**截图 | `vision_parse` → 合并草案（`source=vision`） |
| **③ 对账单截图** | 上传 **交易记录 / 对账单** 截图（**推荐**；可含买入日、金额） | 同 ②，单次最多 **20 张**（**PORT-VISION-BATCH-01**） |

②③ 技术路径相同，**对客区分**是为了引导用户优先提供含 **买入支付金额** 的页面。合并识别后统一走：追问缺失项 → 确认卡 → 写库。

**不做文件解析的理由（产品决项）**

| 项 | 说明 |
|----|------|
| **成本** | PDF/Excel 解析管线、格式兼容、OCR 质量验收成本高 |
| **收益** | 截图 + Vision 已覆盖主流场景；缺字段可 **手输补全** |
| **边界** | 解析结果仍须 **确认卡**，与截图路径终点相同 |
| **识图缺字段** | **对话补全**（买入日、支付金额等）；齐全前不出写库确认卡 |
| **用户要传表格** | 统一话术：「请把页面 **截图** 上传，或直接把持仓 **打字** 发我；暂不支持上传表格/PDF 文件。」 |

#### 8.1.3a 多图上传（对账单 · PORT-VISION-BATCH-01 · P0）

| 项 | 规则 |
|----|------|

### 8.2 修改持仓（快照制 · PORT-SNAPSHOT-01 · P0）

> **本期不做**完整买卖流水表；库内只保留**当前态** `positions` 快照。  
> 调仓含义须同时落在：**对话 + 确认卡变更摘要 + 版本表 `change_summary` 字段**（便于后续解析、审计、Agent 读历史）。

| 规则 | 说明 |
|----|------|
| **同基金多笔买入** | **买入日期不同 → 多行**；同行键 = `fund_code` + `invested_at` |
| **买入支付金额** | 该行买入时实际支付金额（含手续费）；用于成本与持有收益 |
| **卖出 / 清仓** | 自 `positions` **删除**该行；`change_summary.items` 记 `action=remove`；报告可选 **「近期已卖出」**（§8.4.1） |
| **变更流** | 增删改 → Agent 生成 **变更摘要** → **确认卡展示** → `holdings_confirm` 写 **新版本** + 写入 `change_summary` |
| **首版录入** | `change_summary.kind=initial`；`narrative` 如「首次录入持仓」 |
| **读侧** | 面板与 analysis 读 **`is_current=true`**；历史版本保留 `change_summary` 供解析 |

#### 8.2.1 `change_summary`（jsonb · PORT-CHANGE-SUMMARY-01 · P0）

| 中文含义 | 字段 | 类型 | 说明 |
|----------|------|------|------|
| 摘要类型 | `kind` | string | `initial` / `update` |
| 对客叙述 | `narrative` | string | 一段话，如「卖出白酒指数；新增易方达增强回报 3 万」 |
| 结构化条目 | `items[]` | array | 见下表；**建议** 与确认卡条目一一对应 |
| 用户原话 | `user_quote` | string? | 可选 · 保留用户描述调仓的原句 |
| 录入来源 | `source` | string? | `manual` / `vision` / `vision+manual` |

**`items[]` 行**

| 字段 | 类型 | 说明 |
|------|------|------|
| `action` | string | `add` / `remove` / `modify` |
| `fund_code` | string? | — |
| `fund_name` | string? | — |
| `invested_at` | string? | 买入日（add/modify） |
| `detail` | string | 对客可读一句，如「卖出全部份额」「买入支付金额 3 万 → 3.5 万」 |

**版本链（可选 · 便于解析）**：`previous_version_id` → 上一版 `holdings_versions.id`；首版为 `null`。

**对客话术 · 变更确认卡前**

#### 8.3.3 报告章节与对照分析顺序

| 是否存在 **已发布** 对照方案 | 报告必含 | 不含 |
|------------------------------|----------|------|
| **任意**（有持仓即可出报告） | **§一 当前持仓明细**（**PORT-HOLDINGS-IN-REPORT-01**）、概要、总览、结构、单基、收益、风险合规 | — |
| **否**（不对照方案） | 上述 | 配置偏离、再平衡、分批进度、规划书深链 |
| **是**（对照方案） | 上述 + **分批执行进度**（若分批）+ **配置偏离** + **再平衡建议**（大类优先）+ **规划书深链** | — |

**结合方案时的分析顺序（固定）**

1. 读 `holdings_versions` + 选定 **已发布** `allocation_plans`（step=2）  
2. 若 `deploy_mode=分批` → **分批建仓完成度**（§8.3.4）  
3. **大类**配置偏离（当前持仓大类占比 vs `target_allocation`）  
4. **再平衡建议**（沿用方案 `rebalance_rule`，如 ±5%）  
5. **明细基金差异说明** — 持仓基金 ≠ 方案 `detailed_plan` 时：**不强求换基**，按大类给建议  
6. 差异大时 → 报告末 **「与规划书对齐建议」** + 引导复制本报告链接至资产配置更新规划书（**PORT-NO-PLAN-WRITE-01**）

**对客话术 · 报告草稿后、确认发布前（咨询一句 · 非阻断）**

> 对照您的 **「{report_name}」** 方案，大类偏离主要在 **{大类列表}**。  
> 您实际持有的基金与规划明细不完全一致，我已 **按大类** 给出再平衡参考。  
> 您是想先按大类慢慢调整，还是考虑去 **资产配置** 更新方案里的具体基金？

#### 8.3.4 分批建仓完成度（PORT-DEPLOY-01 · P0）

仅当对照方案的 `deploy_mode=分批` 时，在报告 **§五 或并入配置偏离章** 展示：

| 展示项 | 来源 |
|--------|------|
| 计划总规模 / 期数 / 首期与定投安排 | `execution_schedule` |
| **完成度 %** | **已确认买入支付金额合计 ÷ 方案拟定总买入金额 × 100%**（见下表） |
| 完成度一句话 | 如「您已确认买入 **× 万**，方案计划总投入 **× 万**，约完成 **×%**」 |

**公式（定稿）**

| 项 | 规则 |
|----|------|
| **分子** | 当前 `holdings_versions.is_current` 下所有 `holding` 行 `paid_amount` 合计 |
| **分母** | `execution_schedule` 中 **拟定全部买入金额**（与 §7 `total_amount_cny` 或各期加总 **同源**） |
| **不算** | **不按** 时间进度、**不按** 已执行期数单独估算 |
| **一次性** | `deploy_mode=一次性` → **跳过**本段 |

#### 8.3.5 收益测算（PORT-RETURN-01 · PORT-RETURN-02 · P0）

| 指标 | 含义 | 计算 |
|------|------|------|
| **持有收益（绝对值）** | 本笔持仓从买入日到 `as_of_trade_date` 的盈亏金额 | **元** = 最新市值 − 买入支付金额 **+ 持有期现金分红**（见 §8.3.5a） |
| **持有收益率** | 相对成本的回报率 | 持有收益（绝对值）÷ 买入支付金额 |
| **组合合计** | 全持仓加权 | 总成本、总市值、总持有收益（元）、组合收益率 |
| **费率** | 说明口径 | L0/L1 带出申购/赎回/管理费等；正文 **注明** 买入支付金额含申购费 |

**§一 持仓表对客列（P0）**：须同时展示 **持有收益（元）** 与 **持有收益率（%）**；**禁止** 仅展示比例。

| 对客列 | 示例 |
|--------|------|
| 持有收益（元） | **+1,280 元** |
| 持有收益率 | **+4.3%** |

合计行：两列分别汇总（收益率 = 总持有收益元 ÷ 总买入支付金额）。

#### 8.3.5a 分红与基金类型（PORT-RETURN-02 · P0）

> **原则**：不只有「净值涨跌」一种赚法；**现金分红**须单独计入持有收益（绝对值），否则会低估债基、权益类实际回报。

| 基金类型 | 收益怎么体现 | 持有收益计算 |
|----------|--------------|--------------|
| **净值型**（股/债/混/指数等） | 净值涨跌 + 可能 **现金分红** | 市值 − 成本 **+** 持有期现金分红合计 |
| **货币型** | 净值/万份收益累积，**一般无单独现金分红除权** | 市值 − 成本即可；正文可注「货币基金收益已体现在净值」 |
| **分红除权** | 除权日净值下调、现金派发 | 须从 L0 拉分红记录，按 **登记日持仓份额** 汇总现金分红 |

**清仓（卖出）口径（与 PORT-SNAPSHOT-01 一致）**

- 已卖出份额 **不在** §一 当前持仓表；**已实现收益视为随份额带走**，不并入当前表。  
- `change_summary` 含卖出时，报告概要或 §一 末可选一句：「您近期卖出的部分已从当前持仓移除；本表仅反映 **仍持有** 份额的收益。」

#### 8.3.5b 分红与份额调整数据（PORT-RETURN-DIV-01 · P0）

| 优先级 | 来源 | 说明 |
|--------|------|------|
| **1 · L0** | **Tushare**（[基金分红 `fund_div`](https://tushare.pro/document/2?doc_id=120) 等）→ **AKShare** 备用 | 持有期 `[invested_at, as_of_trade_date]` 内 **现金分红**；按 **您确认份额** × 每份分红汇总 `cash_dividend_total` |
| **2 · 降级** | L0 无记录或拉取失败 | 仍算 **市值 − 成本**；脚注「未纳入现金分红」；`metadata.l0_degraded` 含 `dividend_missing` |
| **3 · 禁止** | 默认让用户手工填每笔分红 | **本期不做**；用户可在 **报告确认前对话补充**（§8.3.5c） |

**须纳入测算说明的复杂情况（不保证全自动算准）**

| 情况 | 系统做法 | 局限 |
|------|----------|------|
| **现金分红** | L0 分红记录 + 确认份额汇总 | 登记日与您实际持有份额若有差异，会有误差 |
| **份额调整**（强增/强减/折算） | **以您确认卡上的份额为准**；不自动改写 | 公网历史不一定覆盖您账户内全部调整 |
| **除权除息** | 市值用截止日净值；分红单独加回 | 与代销渠道展示可能差 1～2 个交易日 |
| **货币基金** | 一般不加现金分红项 | 收益已在净值/万份收益 |

#### 8.3.5c 收益为测算值 · 建议用户核对（PORT-RETURN-ESTIMATE-01 · P0）

> **产品定位**：§一/§二 持有收益是 **基于公开数据 + 您确认持仓的测算**，不是代销机构对账单，**不** 作为税务或法律凭证。

| 项 | 规则 |
|----|------|
| **报告必含说明** | §一 章首或 §二 开头 **固定短段**（见下「对客话术」）；**温馨提示** 再收束一句 |
| **默认可发布** | 测算偏差 **不阻断** 报告草稿与发布 |
| **用户补充** | 模式 B 期间，用户可说「某只基分红还有 ×× 元」「份额应是 ××」→ **报告-only 修订** 或 **回持仓确认卡** 改份额后 **重跑分析**（RPT-CHAT-ROUTE-01） |
| **持仓份额/金额** | 以 **您确认保存** 的为准；公网数据 **不覆盖** 您未在对话中确认的变更 |

**对客话术 · §一 章首（测算说明 · 必含）**

**对客话术 · 温馨提示（收益段 · 必含）**

> 本报告持有收益为系统测算，已尽量纳入净值变动与公开分红信息；**份额调整、分红登记日等细节** 可能与账户实际略有差异，请以您的交易记录为准。如需按您的口径修正，可在发布前在对话中说明。

**编码期（PORT-L0-GATHER-01）**：`holdings_nav_gather`（`src/lib/portfolio/holdings-nav-gather.ts`）对持仓去重基金 **`syncFundL0Local(code, { force: true })`** — **每次** `port.rpt.gather.l0` **强制刷新** L0；汇总 `cash_dividend_total`、`dividend_source`；单基失败行「暂无行情」不阻断。`draft-meta.json` 建议 `return_estimate: true` · `l0_degraded[]`。

#### 8.3.6 与资产配置方案比对口径（PORT-ALLOC-COMPARE-01 · P0）

> **原则**：方案约定的是 **目标大类比例 + 计划买入金额**；持仓录入的是 **买入支付金额**。二者同属 **「投入了多少钱」** 维度，可以比对。**最新市值** 含净值涨跌，只用于 **§二 收益**，**禁止** 用于 §三 结构分布、§五 配置偏离、§六 再平衡判断。

| 用途 | 持仓侧口径 | 方案侧口径 |
|------|------------|------------|
| **§三 结构分布** | 各大类 `paid_amount` 合计及占比 | —（无方案时仅展示持仓成本结构） |
| **§五 配置偏离** | 各大类 **买入支付金额占比** | `target_allocation` 目标 **%** |
| **§五 分批进度** | 持仓 `paid_amount` 合计 | 方案计划应到位金额（约） |
| **§六 再平衡** | 基于 §五 成本口径偏离 | `rebalance_rule` |
| **§二 持有收益** | **可用** 最新市值 | **不参与** |

报告须在 §五（或结构相关章）写清一句对客说明：**「与方案比对统一按买入支付金额，不使用最新市值。」**

- 再平衡与「持仓录入时间」**无关**；仅取决于是否存在 **已发布** 对照方案（§8.3.1）  
- **禁止** 用未完善 `goal_constraint_id` 上的方案做偏离/再平衡

### 8.5 分析时效提示（与再平衡无关）

进入 **`portfolio` 场景**时，若存在历史持仓分析报告，主区顶部 **非阻断** 提示条：

| 项 | 说明 |
|----|------|
| **{N}** | 今天 − 最近一篇 `report_type=portfolio` 的 `generated_at`（自然日） |
