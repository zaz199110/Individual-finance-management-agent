### 4.2.1 自然日调度 × 最近交易日（SCH-01）

> **原则**：**什么时候跑**看用户 **本地日历 + 本地钟点**；**报告用哪天的行情**看 **沪深交易日历**（`trading_calendar`），不在非交易日伪造净值。

#### 4.2.1a 两个「时间」分别是什么？（对客 / 编码必读）

| 概念 | 用什么时区 / 日历 | 例子 |
|------|-------------------|------|
| **① 何时触发** | 用户电脑 **系统本地时区** + **本地日历** | 设「每周三 09:00」→ 本机周三 9:00 到点尝试执行 |
| **② 数据截止日 `as_of_trade_date`** | 触发瞬间转为 **北京时间（Asia/Shanghai）日历日** 为锚，在 `trading_calendar` 中取 **该日及之前最近一个 `is_open=true` 的日期** | 本机周日 09:00 触发 → 锚点可能是北京时间周日 → 截止日为上交所最近周五 |

**为何不能全用本地时区算交易日？** 国内公募基金净值、沪深交易日历均按 **国内交易所** 披露；用户在海外设定时，仍应得到与国内一致的「截至哪个交易日」，而不是按纽约周末误判。

**为何触发不用北京时间？** 用户期望「早上 9 点跑」是 **自己生活中的 9 点**；若强制北京时间，海外用户体验失真。故 **触发本地、截止日按北京 + 国内日历**（SCH-07）。

**不用 LLM 联网查交易日**：日历须来自 **Tushare / AKShare** 结构化接口（§2.8.6 · §4.2.6）；LLM 检索不稳定且不可审计。

**SSE 与 SZSE（本期存储口径 · SCH-13）**：A 股休市日在上交所（SSE）与深交所（SZSE）**一致**；国内公募净值披露按 **沪深 A 股交易日** 口径。本期 **`trading_calendar` 仅拉取、仅存 `exchange='SSE'`**，**不重复**拉 SZSE；编码计算 `as_of_trade_date` 时视同 **沪深交易日历**。

**两步分离**

| 步骤 | 规则 |
|------|------|
| **① 调度触发** | 本地日历到点即 **尝试** 执行（含周六、日、法定假） |
| **② 数据截止** | 按 4.2.1a ② 计算 `as_of_trade_date` 后拉 L0 行情 |

**错过触发（SCH-08）**：应用未运行导致未触发 → **不补跑**、**不写日志**；用户可在持仓 Tab **手动分析** 或等下一周期。

**交易日历刷新**：进入本页检查 **当前公历年**（建议含 **次年**）是否已在 `trading_calendar`；缺失则拉取（§4.2.6 · SCH-11）。MVP 已预置 **`seed/trading_calendar_2026.sql`**。

**报告页眉（对客 · 与 §8.4.1 一致）**

- 须写明：**「数据截至 {as_of_trade_date}（最近交易日）」**
- 「我的报告 · 持仓分析」列表副行可标 **「定时生成」**（`trigger_source=scheduled`）
- **QDII 占比提示（SCH-13）**：持仓中 **QDII 类基金合计市值占比 ≥30%** 时，概要/页眉 **追加**：「部分 QDII 基金净值披露节奏可能与 A 股交易日不同，本报告已取各持仓共同可用的最近截止日。」——**不换日历**，仅增强说明

### 4.2.2 执行链路（SCH-06 · 后台 · P0）

| 项 | 规格 |
|----|------|
| **调度器** | Harness **s14**（§4.2.6）；与 s13 后台执行器共用 |
| **每次触发** | **新建**一条 `conversation_type=portfolio` 对话；**后台**跑 Harness，**不**切模式 B |
| **用户可见** | 侧栏历史 **多一条** portfolio 对话；标题 **`定时持仓分析 · {YYYY-MM-DD}`**（SCH-12） |
| **发布** | Verify 通过 → **直接** `report_publish`（RPT-SCHED-01）；**无**确认卡 |
| **日志** | 写入 `scheduled_job_runs`；成功填 `report_index_id` + `conversation_id` |
| **失败** | `status=failed` + `failure_reason`；`consecutive_failures += 1`；**≥3 次连续 failed** → `enabled=false` + Toast「定时持仓分析已因连续失败自动关闭」；**skipped / success** 将计数 **归零** |

#### 4.2.2a Harness 等价路径（定稿 · coding-ready）

> **定时 = 手动持仓分析同一路径**，仅下列差异；**不另写**定时专用 Handler / Skill。

| 步骤 | 手动（§8.4） | 定时（RPT-SCHED-01） |
|------|--------------|----------------------|
| 入口 | 用户对话 · `portfolio` Tab | s14 后台 · 新建 `portfolio` 对话 |
| 分析 | 同一 **portfolio Handler** + 同一 Tool 链（`holdings_read` → `report_draft` …） | **相同** |
| Verify | 跑 Verify | **相同** |
| 发布后 | 模式 B · **确认卡** → 用户点发布 | **跳过**确认卡 · Verify 通过后 **直接** `report_publish` |
| 元数据 | `trigger_source=manual` | `trigger_source=scheduled` + `as_of_trade_date`（§4.2.1） |
| 报告章节 | PORT-01（§8.4.1） | **相同** |

**注入 prompt（最小约定）**：后台会话首条可等价于用户说 **「请基于当前持仓生成持仓分析报告」**；具体字面编码期与手动「重新分析」对齐即可，**不改变** Tool 编排。

#### 4.2.2b Verify 与人工（定稿）

> **Verify** = Harness **自动校验层**（HARNESS §11），**不是**用户点「确认发布」。与 **人工确认卡** 是两层。

| 层 | 是什么 | 谁执行 | 定时任务 |
|----|--------|--------|----------|
| **Verify** | 报告草稿写完后 **程序校验**：合规段、图表、数据源、`mermaid` 可渲染等 | **机器** | **必须跑**；**通过**才 `report_publish` |
| **人工确认** | 模式 B · **确认卡**「是否发布报告」 | **用户** | **一律跳过** · Verify 通过后 **直接发布**（RPT-SCHED-01） |

| 结果 | 定时任务行为 |
|------|--------------|
| **Verify 通过** | 直接发布 → `scheduled_job_runs.status=success` |
| **Verify 未通过**（机器校验失败） | **不发布** → `status=failed` + `failure_reason`（计入连续失败） |
| **若链路某步「必须等人」才能继续**（不应出现在定时路径） | **不等待** → 记 **`skipped`** · `skip_reason=需人工确认，定时任务已跳过`；**不**挂起任务 |

**原则**：定时路径 **禁止**进入模式 B、**禁止**弹出确认卡；凡依赖用户点击才能往下走的步骤，**默认跳过本次**，写日志，**不**无限等待。

| 规则 | 说明 |
|------|------|
| **手动优先** | 用户正在跑 **手动** 持仓分析（`workflow_locks.holder` 或同会话 running job）→ 定时触发 **排队或延后**至手动结束（实现取 **跳过本次** 更简单） |
| **同日去重** | 若 **同一自然日（本地日历）** 已存在 **`trigger_source=manual`** 的 `report_index`（`report_type=portfolio`）→ 定时触发记 **`skipped`** · `skip_reason=当日已有手动分析，行情相同不再重复生成` |
| **不做「立即执行一次」**（SCH-02） | 临时分析走持仓 Tab 手动触发 |

### 4.2.6 s14 调度轮询（SCH-14 · P0）

> **桌面本地 App**：定时 **只在客户端进程存活时** 才可能触发；**不是**云端 cron。用户 **不必打开「定时持仓分析」页**，但 **App 须已启动**（含最小化到托盘，若产品支持托盘常驻）。

| 场景 | 是否触发 |
|------|----------|
| App **已启动**（前台 / 后台 / 托盘） | s14 每 60s tick → 到点可跑 |
| App **完全退出** | **不触发** · **不补跑** · **不写日志**（SCH-08） |
| 仅打开浏览器访问（若未来有 Web 版） | **本期无**；本期假定 **Electron/Tauri 桌面进程** |

#### 定稿：每 **60 秒** 扫一次

| 项 | 规格 |
|----|------|
| **轮询间隔** | **`60_000 ms`**（每 1 分钟 tick 一次） |
| **为何不用更细（如 10s）** | 用户配置精度为 **分钟**（`HH:mm`）；秒级无意义，且空转耗 CPU |
| **为何不用更粗（如 5min）** | 5 分钟 tick 可能 **跨过** 09:00 窗口且无法对齐「这一分钟」（如 08:58 → 09:03 才检查），在**不补跑**前提下易 **漏跑** |
| **实际触发误差** | 设 09:00 → 最晚约在 **09:00:59** 当次 tick 启动（**≤1 分钟** 延迟，可接受） |

#### 每次 tick 判定（本地时区）

1. `scheduled_jobs.enabled = true` 且存在当前持仓  
2. **今天（本地日历）** 是否命中 `schedule_kind` + `schedule_days`（每周星期 / 每月日期 · 含短月规则）  
3. **当前本地时间** `HH:mm` **等于** `run_at_time`（只比到分钟）  
4. **本分钟未跑过**：同一 `job_id` 在 **同一本地分钟** 无 `success`/`failed`/`skipped` 的 `scheduled_job_runs`（防重复）  
5. 通过 §4.2.3 **并发 / 同日去重** → 否则记 `skipped`  
6. 通过则：新建 portfolio 对话 → 后台 Harness（§4.2.2a）→ 写 `scheduled_job_runs`

#### 与「算下次唤醒点」的对比（本期不做）

| 方案 | 优点 | 缺点 |
|------|------|------|
| **60s 轮询（本期）** | 实现简单；改配置不用重算 timer；多选星期/日逻辑集中在一处 | 最多 1 分钟延迟 |
| **sleep 到下一触发点** | 精准到秒 | 改 `run_at_time`/频率要重算；App 休眠唤醒复杂 |

#### 实现锚点

- 模块：`harness/cron/` 或 `s14_cron_scheduler/`（HARNESS §8d2）  
- App 启动时 `start()`；**仅主进程 / 含 Harness 的后台 worker** 跑 s14（避免多实例双跑）  
- tick 内 **串行**：同一时刻最多 **1** 个定时持仓任务在跑（与 `workflow_locks` · portfolio 一致）

### 4.2.6 字段规格（Supabase）

> 页面规格见 §4.2.0；**禁止**用 LLM 联网猜交易日（SCH-01）。

#### 4.2.6a `scheduled_jobs`（配置 · 单行 seed）

| 中文含义 | 字段名称 | 字段类型 | 字段长度 | 是否必填 | 字段校验 | 值的相关说明 |
|----------|----------|----------|----------|----------|----------|--------------|
| 主键 | `id` | uuid PK | uuid | 系统 | — | — |
| 任务类型 | `job_type` | text **UNIQUE** | 固定 `portfolio` | 是 | 全局 1 行 SCH-06 | — |
| 是否启用 | `enabled` | boolean | — | 是 | 默认 false | 清空持仓关 SCH-09 |
| 周期类型 | `schedule_kind` | text? | — | 条件（启用时） | 二选一 | `weekly` / `monthly` |
| 执行日 | `schedule_days` | int[]? | 1–7 或 1–31 | 条件 | 与 kind 匹配 | — |
| 执行时刻 | `run_at_time` | text | `HH:mm` | 是 | 默认 09:00 | 本地时区 |
| 连续失败次数 | `consecutive_failures` | int | ≥0 | 是 | ≥3 自动 disabled SCH-12 | — |
| 更新时间 | `updated_at` | timestamptz | — | 系统 | 配置变更时 | — |
| 上次触发 | `last_run_at` | timestamptz? | — | 否 | — | 含 skip/fail |

**Migration seed（SCH-06）**：INSERT 1 行 `job_type=portfolio` · `enabled=false` · `run_at_time='09:00'`（幂等 `ON CONFLICT DO NOTHING`）。

#### 4.2.6b `scheduled_job_runs`（任务日志）

| 中文含义 | 字段名称 | 字段类型 | 字段长度 | 是否必填 | 字段校验 | 值的相关说明 |
|----------|----------|----------|----------|----------|----------|--------------|
| 主键 | `id` | uuid PK | uuid | 系统 | — | — |
| 所属配置 | `job_id` | uuid FK | — | 是 | FK | → `scheduled_jobs.id` |
| 触发时刻 | `triggered_at` | timestamptz | — | 是 | — | UTC 存储 |
| 执行状态 | `status` | text | — | 是 | 三选一 | `success` / `failed` / `skipped` |
| 关联报告 | `report_index_id` | uuid? FK | — | 否 | FK | 成功时 |
| 后台对话 | `conversation_id` | uuid? FK | — | 否 | FK | 新建的 portfolio 对话 |
| 失败原因 | `failure_reason` | text? | ≤500 字 | 否 | — | — |
| 跳过原因 | `skip_reason` | text? | ≤200 字 | 否 | — | 如 `same_day_manual_exists` |
| 行情截止日 | `as_of_trade_date` | date? | — | 否 | — | 最近交易日 |

索引：`scheduled_job_runs(job_id, triggered_at DESC)`。

#### 4.2.6c `trading_calendar`（交易日缓存）

| 中文含义 | 字段名称 | 字段类型 | 字段长度 | 是否必填 | 字段校验 | 值的相关说明 |
|----------|----------|----------|----------|----------|----------|--------------|
| 日历日 | `cal_date` | date PK | — | 是 | 主键 | 自然日 |
| 交易所 | `exchange` | text | 固定 `SSE` | 是 | — | SCH-01 |
| 是否开市 | `is_open` | boolean | — | 是 | — | 沪深是否开市 |
| 公历年份 | `year` | int | 4 位 | 是 | — | 按年检查缓存 |
| 拉取时间 | `fetched_at` | timestamptz | — | 是 | — | 最近同步 |
| 数据来源 | `source` | text | — | 是 | 二选一 | `tushare` / `akshare` |

**刷新策略（SCH-11）**：进入 `/scheduled-jobs` 缺当前年数据 → 异步拉取（建议含次年）；s14 执行前缺数据 → 同步拉取；数据源 Tushare → AKShare；MVP 种子 `seed/trading_calendar_2026.sql`。

**清空持仓联动（SCH-09）**：`holdings_confirm` 清空成功后同事务 `UPDATE scheduled_jobs SET enabled=false` + Toast（§4.2.0d）。
