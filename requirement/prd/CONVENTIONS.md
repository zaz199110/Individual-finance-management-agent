# PRD 撰写规范

**一份事实、一处详文**：同一规则只写在一个模块里；别处用 `§x.y` 或链接，不抄正文。

---

## 1. MERGE 三原则

| 原则 | 做法 |
|------|------|
| **一处详文** | 两处写同一件事 → 留详文，另一处改一行链接 |
| **Hub 不重复** | Hub（02/04/05/09）只保留：模块说明、子块对照表、决项索引、子文档链接 |
| **决项一行** | 已拍板 → [appendix-d](./appendix-d-decisions.md) 一行 + 正文一句；名词 → [GLOSSARY](./GLOSSARY.md) |

---

## 2. 怎么写

**阅读顺序（对人）**：模块说明（做什么 / 入口 / Done / 不做）→ **场景 → 流程 → 功能** → 字段规格（给开发）→ Command / REST。

- 拆分顺序：**场景 → 流程 → 功能** 用于 **选完场景后的各目标节**（如 §6.2.3–§6.2.7）；整章流程概览、选场景、通则、写库另立节，不强行套三层标题。示例 → [§6.2](./06-profile.md)。

- **中文主句** + 同段补编码锚点（表名、字段、`XX-01`），不做人话版/Agent 版双份。
- 术语首次出现：中文为主，括号补字段名；**定义只写在 GLOSSARY**。
- 结构：**场景 → 做法 → 边界 → 不做**；再写字段/API。
- **表字段表**（Supabase / 本地索引 / JSON 契约）须写全的字段，统一 **七列**：

| 中文含义 | 字段名称 | 字段类型 | 字段长度 | 是否必填 | 字段校验 | 值的相关说明 |
|----------|----------|----------|----------|----------|----------|--------------|

| 列 | 写什么 |
|----|--------|
| **中文含义** | 业务可读名 |
| **字段名称** | 库列名 / JSON 键 / API 键（反引号） |
| **字段类型** | SQL 类型、`jsonb` 内类型、枚举集合 |
| **字段长度** | 字符上限、数值范围、数组元素上限、精度；无则 `—` |
| **是否必填** | `是` / `否` / `条件（…）` / `系统`（库自动生成） |
| **字段校验** | 枚举、Hook、公式、跨字段规则（可写 `§6.1.3` / Hook 编号） |
| **值的相关说明** | 取值含义、默认值、外键/联动、下游只读 |

**枚举可读性**：取值列用 **`code`（中文）** 用 `/` 分隔；**禁止**在单元格里堆旧式一行写法。

**从简**：纯系统列可合并为一行，后四列写 `—`。

**对客问卷**（§6 需求梳理、§7 方案等）：**字段七列只在对应模块「字段规格」写一次**；问卷用普通人能复制的话术，**禁止**堆字段名、Hook 编号或库表术语。问卷同步到 `skills/**/questionnaire.*.zh.md`。

**对客确认卡**（P-04）：**中文含义 + 结论**；字段型卡片以字段表为唯一清单，**不**另写卡内字段副本。通则 → [05-chat-shared §5.3.10b](./05-chat-shared.md)；Mock → `skills/shared/confirm_card.mock.zh.md`。

**对客报告草稿**（profile / plan / fund 等）：PRD 定 **必含章节 + 数据绑定 + Verify**；Agent 入口 → `skills/**/report.template*.zh.md`；**详文** → `requirement/docs/samples/*-report-spec.md`；**对客 Mock** → `*-report-sample.md`。  
**基金解读 / 投资规划书** 另须 **` ```echarts `** 围栏 + Preview **JSON 解析**（FK-18-EC / PL-REPORT-EC）；投资需求报告 **可选** 0～2 图（PROFILE-VISUAL-01 · 有触发才画）；持仓分析报告 **少而精** A **2** / B **3** 图（PORT-VISUAL-01 · 表精确、图亮眼）。

**字段权威位置**：各 **模块 PRD 的「字段规格」节**；总索引 → [03-data-architecture §3.4](./03-data-architecture.md)。Hub 与子文档 **不重复** 七列表。

- 文件头统一：`> [← PRD 索引](../PRD.md) · **N. 标题**`（版本、规范链接只在 [PRD.md](../PRD.md)）。

## 3. 模块文件头（`### 模块说明`）

| 项 | 写什么 |
|----|--------|
| **做什么** | 用户要完成的事（一句） |
| **入口** | 对客入口 |
| **完成标志** | 可见 Done；中间步骤须标明 |
| **不做** | 本期不做（1–3 条） |
| **依赖** | 链到其他 §，不写重复规则 |
| **编码锚点** | 可选：主表 / API 前缀 / 功能 ID |

## 5. § 编号

0–9 按模块；Hub 子块仍用 §2.2 / §4.1 / §5.3 / §9.1 等。**跨文件引用须带 §x.y**。同一文件内 **按出现顺序递增**，避免 3.3.2 写在 3.3.4 后面、3.4.0 写在 3.4.3 后面这类乱序。

## 6. 改 PRD 检查

1. 是否落在 **最小模块**？  
2. 是否与他处重复？→ 删重复，改链接  
3. Hub 是否只有对照表？  
4. 新决项是否只在 appendix-d 增一行？  
5. 字段是否只在 **一处「字段规格」**？  
6. 是否对照 HARNESS §13？
